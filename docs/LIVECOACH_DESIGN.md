# LiveCoach — реалтайм-тренер во время тренировки

Статус: **проектирование**, реализация — следующий большой блок.
Дата фиксации: апрель 2026.

## 1. Принципиальный сдвиг концепции

**Раньше:** ProgressionEngine работал на уровне сессии — после всей тренировки разбирал прошедшее, давал рекомендации на **следующую тренировку**.

**Теперь:** LiveCoach работает на уровне **подхода** — после каждого сета анализирует, реагирует, корректирует следующий подход **прямо сейчас, в этой же тренировке**.

ProgressionEngine остаётся для долгосрочной картины (между тренировками). LiveCoach — для реалтайма внутри тренировки.

## 2. Главная философия

**НЕ «следуй плану»** — план это отправная точка, не контракт.
**ДА «адаптируем план под комфорт»** — корректируем по самочувствию и накопленной статистике.

Конкретно:
- Много RIR на запланированном весе ≠ лентяй. Это сигнал «вес подобран неправильно». Предложить +2.5 кг
- Мало повторов ≠ провал. Может день не зашёл, может усталость. Предложить -2.5 кг или ничего
- Решения опираются на **накопленную историю + текущее самочувствие**, не на одну сессию

## 3. UX-флоу

Базовый сценарий тренировки:

```
1. Перед тренировкой → быстрый picker самочувствия (😴/🙂/💪)
2. Открыли упражнение
3. Подход 1:
   - Ввёл повторы + RIR (RIR обязательный)
   - Тапнул "✓ выполнено"
   - LiveCoach анализирует
   - Реакция (если есть отклонение):
       "RIR 4 → лёгко прошло. -2.5 кг попробуем?"
       [Да] [Нет, оставить]
   - Запускается таймер отдыха
   - Во время отдыха — микро-контент из пула
4. Подход 2: с (возможно) скорректированным весом
5. Между упражнениями → опционально picker энергии
6. Подход N → ...
7. После тренировки → анкета:
   - Энергия (1-5)
   - Сон (1-5)
   - Стресс (1-5)
   - Настроение (опционально)
8. AI-разбор по итогам (старая логика, но с учётом самочувствия)
```

## 4. Сбор самочувствия

**Перед тренировкой** (быстрый picker, 1 тап):
- 😴 Уставший
- 🙂 Норма
- 💪 Бодрый

**Между упражнениями** (опционально, не каждый раз):
- "Как ощущения?" — тот же picker
- Появляется только если сильное отклонение в предыдущем упражнении или прошло >3 упражнений

**После тренировки** (обязательная анкета):
- Энергия 1-5
- Сон 1-5 (как спал прошлой ночью)
- Стресс 1-5
- Настроение/мотивация (опционально)

Эти данные:
- Сохраняются в новую таблицу `wellness_log`
- Используются LiveCoach как контекст в моменте
- Используются ProgressionEngine как фильтр («3 недели стресс высокий → не наказывать за регресс»)

## 5. LiveCoach — ядро движка

### Вход

```kotlin
data class SetContext(
    val setNumber: Int,                      // какой по счёту
    val totalSets: Int,                      // всего в упражнении
    val targetReps: IntRange,                // план (например 6..10)
    val targetRir: Int,                      // целевой RIR (от цели программы)
    val plannedWeight: Double,               // плановый вес
    val actualReps: Int,                     // что сделал
    val actualRir: Int,                      // субъективная оценка
    val actualWeight: Double,                // фактический вес
    val previousSetsThisExercise: List<SetLog>,    // подходы этого упражнения
    val historicalPerformance: ExerciseHistory?,    // последние сессии этого упражнения
    val currentWellness: WellnessSnapshot?,        // самочувствие сейчас
    val exercise: Exercise
)

data class WellnessSnapshot(
    val preWorkoutEnergy: Energy?,           // FATIGUED/NORMAL/ENERGIZED
    val betweenExerciseEnergy: Energy?,      // если был ввод
    val sevenDayAvgEnergy: Double?,          // тренд
    val sevenDayAvgSleep: Double?
)
```

### Выход

```kotlin
data class SetReaction(
    val verdict: SetVerdict,                 // ON_TARGET / EASY / HEAVY / TOO_HEAVY / TOO_EASY
    val confidence: Confidence,
    val shortMessage: String,                // что сказать юзеру (1-2 предл)
    val nextSetSuggestion: NextSetTip?,      // что делать со след. подходом
    val restSeconds: Int,                    // может корректироваться
    val coachContent: CoachContent?          // что показать в отдыхе
)

enum class SetVerdict {
    ON_TARGET,        // в плане, RIR в норме — молчаливая галочка
    EASY,             // лёгко далось (RIR > цели)
    HEAVY,            // тяжело (RIR < цели), но в диапазоне повторов
    TOO_EASY,         // RIR сильно выше + повторы выше плана
    TOO_HEAVY,        // RIR=0 + повторы ниже плана
    UNUSUAL,          // странное сочетание (например high reps + low RIR)
    INSUFFICIENT      // RIR не ввёл — слабая реакция
}

data class NextSetTip(
    val type: WeightAdjustment,              // INCREASE / DECREASE / KEEP / MORE_REST
    val newWeight: Double?,                  // если меняем вес
    val newRestSeconds: Int?,                // если меняем отдых
    val rationale: String,                   // почему
    val requiresConfirmation: Boolean = true // юзер выбирает Да/Нет
)
```

### Решающая логика — упрощённый алгоритм

```
ШАГ 1. Сравнение с планом + RIR
  - повторы в плане + RIR в цели ±1 → ON_TARGET, молчаливая галочка
  - RIR > цели на 2+ + повторы в плане → EASY
  - RIR < цели на 2+ + повторы ниже плана → HEAVY/TOO_HEAVY
  
ШАГ 2. Контекстная корректировка через wellness
  - Если HEAVY и юзер сказал "уставший" перед тренировкой → 
    "тяжело даётся, день не зашёл — оставь как есть"
  - Если HEAVY и юзер свежий → 
    "тяжело сегодня, может вес высокий — попробуй -2.5"
    
ШАГ 3. Контекстная корректировка через историю
  - Если EASY впервые на этом весе → 
    "легко, может новый PR? оставь, посмотрим"
  - Если EASY стабильно 2+ подхода → 
    "явный запас, +2.5 кг попробуем"
    
ШАГ 4. Решение по следующему подходу
  - При EASY/TOO_EASY → INCREASE +2.5 (или +шаг по таблице)
  - При HEAVY/TOO_HEAVY → DECREASE -2.5
  - При ON_TARGET → KEEP
  - При UNUSUAL → KEEP, но коуч-контент с уточняющим вопросом
```

### Архитектура

`domain/livecoach/LiveCoach.kt` — детерминированный движок (без AI).
Чистый Kotlin, легко юнит-тестируется.

## 6. Корректировка веса в моменте — UX

**Подход: предлагаем выбор.**

После завершения подхода с отклонением:

```
┌───────────────────────────────┐
│  RIR 4 на 60×8                │
│  Лёгко прошло — есть запас    │
│                               │
│  Поднять до 62.5 кг?          │
│                               │
│  [Да, добавить]   [Оставить]  │
└───────────────────────────────┘
```

При тапе "Да" — следующий подход открывается с подставленным новым весом.
При тапе "Оставить" — следующий подход с тем же весом.
При игноре (просто скролл вниз) — равно "Оставить".

Корректировка применяется **только к следующему подходу**, не ко всем оставшимся. Если повторно EASY — снова предлагаем.

## 7. RIR — обязательный

В новом WorkoutScreen:
- Поле RIR в фокусе после ввода повторов
- Кнопка "✓ выполнено" заблокирована пока RIR не введён
- Есть кнопка-выход "не помню" — RIR=null, LiveCoach реагирует с низкой уверенностью

Onboarding или первая тренировка должна научить юзера что такое RIR. Можно сделать tutorial-tooltip при первом вводе.

## 8. Контентный пул

### Структура

Для каждого упражнения хранится JSON-пул:

```json
{
  "exerciseId": "bench-press",
  "name": "Жим штанги лёжа",
  "content": [
    {
      "type": "TECHNIQUE",
      "tag": "lopatki",
      "text": "Своди лопатки и держи их сведёнными весь подход — это стабилизация плечевого сустава"
    },
    {
      "type": "MISTAKE",
      "tag": "elbows-flare",
      "text": "Локти не должны разлетаться в стороны на 90° — это нагрузка на плечо. Угол ~45°"
    },
    {
      "type": "MOTIVATION",
      "tag": "control",
      "text": "Не гонись за весом — гонись за чистым повтором. Качественный 60×8 лучше грязного 65×8"
    },
    {
      "type": "FACT",
      "tag": "biomechanics",
      "text": "При жиме лёжа большая грудная мышца берёт ~50% нагрузки, остальное — трицепс и плечи"
    }
  ]
}
```

Типы:
- TECHNIQUE — техника движения
- MISTAKE — типичная ошибка
- MOTIVATION — установка мышления (не гнать за весом, контроль и т.п.)
- FACT — научный/анатомический факт
- TIP — конкретный совет (дыхание, темп и т.п.)

20 единиц на упражнение × 50 упражнений = 1000 текстов.

### Генерация

**Один раз через AI batch.**

Делаем скрипт `tools/generateContentPool.kt` (CLI или один-shot Kotlin):
1. Берёт список упражнений из preset-программы + типичные движения
2. Для каждого упражнения вызывает AI с промптом «дай 20 текстов разных типов про это упражнение»
3. Сохраняет в `app/src/main/assets/coach_content/{exerciseId}.json`
4. Юзер (ты) редактирует топ-10 упражнений вручную, остальное оставляет как AI выдал

При сборке APK эти JSON попадают в bundle. На устройстве — читаются из assets.

Стоимость генерации: 50 упражнений × ~$0.05 (4o-mini) = ~$2.50 один раз.

### Алгоритм выбора во время отдыха

При начале таймера отдыха:
1. Берём пул для текущего упражнения
2. Фильтруем те что показывали в последние 3-4 недели (нужна история показов в БД)
3. Случайно выбираем тип (с весами: TECHNIQUE 30%, MISTAKE 25%, MOTIVATION 20%, FACT 15%, TIP 10%)
4. Случайно выбираем единицу из этого типа
5. Записываем в историю показов с timestamp

Раз в 3-4 недели повтор — приемлемо. Главное чтобы не было «3 раза подряд про лопатки».

## 9. Молчаливая галочка vs коммуникация

При ON_TARGET — короткая визуальная коммуникация без отвлекающих карточек:
- Подход чекбокс становится зелёным с галочкой
- Под ним 1 строка: "В плане, RIR в норме" (мелким шрифтом)
- Сразу запускается таймер отдыха

Большая карточка реакции — только при отклонениях (EASY/HEAVY/TOO_*).

## 10. Что меняется в WorkoutScreen — UI

### Сейчас
- Список упражнений с подходами
- На каждом подходе: №, Вес, Повторы, RIR, чекбокс
- При тапе чекбокса — стартует таймер 90с

### После переработки

**Активный подход подсвечен.** Только текущий подход в фокусе, остальные свёрнуты.

**Когда подход завершается** (тап ✓):
1. Анимация успеха (галочка зелёная)
2. Карточка реакции (если есть отклонение) — слайдом снизу
3. Таймер отдыха в большом формате
4. Под таймером — карточка коуч-контента
5. Когда таймер закончился — фокус на следующий подход

**Между упражнениями** (закончил все подходы):
1. Опционально — picker «Как ощущения?»
2. Кнопка «Перейти к следующему» (с автопереходом через 5 сек)

## 12. Обновление AI-разбора с учётом wellness

Когда дойдём до полного цикла — `ReviewRepositoryImpl` тоже знает про wellness:

```
Передавать в AI-промпт:
- Самочувствие до тренировки
- Среднее самочувствие за последние 7 дней
- Stress-индекс
- Скорректированные подходы (юзер принял предложение или отклонил)

AI учитывает в выводах:
- "Ты дал зелёный свет +2.5 на жиме — отлично"
- "Тяжело сегодня, но недавно был стресс — нормально, не ставим в минус"
```

## 13. План реализации (примерная очерёдность)

1. **Контентный пул** — генерация через AI batch + структура (1-2 дня)
   - Скрипт генерации
   - JSON-формат
   - Загрузка из assets
2. **Wellness таблицы + сбор данных** (1 день)
   - Pre-workout picker
   - Post-workout анкета
   - Сохранение в БД
3. **LiveCoach core** — детерминированный движок (1-2 дня)
   - Модели Reaction/Tip/Verdict
   - Решающая логика
   - Юнит-тесты на 15-20 кейсов
4. **WorkoutScreen рефакторинг** — большой UI блок (2-3 дня)
   - RIR обязательный с фокусом
   - Карточка реакции с предложением
   - Контентный таймер отдыха
   - Активный подход подсвечен
5. **Интеграция wellness в AI-разбор** (полдня)
6. **Picker между упражнениями** (опционально, полдня)

Итого: ~5-7 дней работы.

## 14. Что переносится из текущей системы

- ProgressionEngine — остаётся для долгосрочного разбора (между тренировками)
- SanityValidator — применяется и к LiveCoach предложениям тоже
- AI-разбор после тренировки — остаётся, расширяется wellness-данными
- 4%-cap, шаги по диапазонам, целевой RIR по цели — переиспользуем для LiveCoach

## 15. Что НЕ делаем сразу

- AI в моменте подхода (Режим 1: только локальный движок для альфы)
- Голосовые подсказки во время отдыха
- Видео в коуч-контенте
- Автоматическая подмена веса (только с подтверждением)
- Адаптация программы on-the-fly целиком (только подход за подходом)

Отложено до следующих этапов.
